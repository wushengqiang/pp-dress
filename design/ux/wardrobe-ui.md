## Purpose & Player Need

衣橱 UI 的目的，是让玩家在进入每日场景前，快速、安静地完成今天的穿搭选择，并在这个过程中持续感到“这是我的选择”。它要服务的核心需要有三层：第一，清晰浏览已解锁与未解锁的服装，方便比较与试配；第二，用拖拽为主、点击为备的交互，让鼠标和触摸都能自然完成换装；第三，让每一次操作都温柔、有反馈，但不引入评分、失败或压力。

如果这个界面不存在，玩家就无法把“今天穿什么”变成一个轻松、可重复的日常仪式；如果它难用，整个换装体验就会从“表达”退化成“管理库存”。因此，衣橱 UI 必须始终把角色与穿搭放在视觉和情绪中心，自己退到安静的边缘，只负责把选择变得简单、明确、令人满足。

---

## Player Context on Arrival

玩家进入衣橱 UI 时，情绪应当是轻松、没有压力的。这里不假设玩家正赶时间，也不假设她需要做复杂决策；她只是刚被游戏流程自然带到“今天该怎么穿”的步骤上。

这个界面不是玩家主动搜索出来的工具页，而是游戏日常循环的一部分：系统自动把她带到这里，让她顺着当前一天的节奏开始搭配。玩家到达时应该已经处在“准备表达自己”的状态，而不是“处理任务”的状态。

---

## Navigation Position

衣橱 UI 位于游戏主循环的中段，路径是 `主菜单 → 开始今天 → 衣橱 UI`。它不是全局常驻的顶层页面，而是一个由当天流程自动进入的上下文界面：玩家先从主菜单开始今天，再进入衣橱完成搭配，之后才进入每日场景。

这个位置意味着衣橱 UI 既要承接主菜单的“开始今天”动作，也要为后续的每日场景准备好穿搭结果。它的任务不是导航分发，而是在当天流程中把“开始”自然过渡到“选择穿搭”。

## Entry & Exit Points

| Entry Source | Trigger | Player carries this context |
|---|---|---|
| 主菜单 | 玩家点击“开始今天” | 当前天数、当日流程开始、准备进入衣橱搭配 |

| Exit Destination | Trigger | Notes |
|---|---|---|
| 主菜单 | 玩家按回退并确认离开 | 若尚未确认穿搭，先显示轻量确认提示；玩家确认后返回主菜单并放弃本次临时编辑 |
| 每日场景 | 玩家确认穿搭 | 离开衣橱后进入当天场景，当前搭配继续生效 |
| 主菜单 | 玩家主动离开当前流程 | 真正的游戏退出不在衣橱内处理，由主菜单统一承接 |

## Layout Specification

### Information Hierarchy

1. 服装本体
2. 是否解锁
3. 分类
4. 选中 / 已装备状态
5. 可交互提示（拖拽、点击）

### Layout Zones

核心布局原则：角色居中，衣橱 UI 后退；桌面端和移动端使用不同区域安排，但都保持角色与当前穿搭为视觉焦点。

**桌面端（≥768px）**
- 顶部条：返回、当天信息、确认穿搭
- 左侧类目列：固定显示 6 个类目标签，禁用类目保留位置并显示锁定态
- 中央主区：角色预览与当前穿搭展示
- 底部网格：当前分类的服装网格 / 横向滚动列表
- 底部或角色侧边：当前选中物品与提示状态

**移动端（<768px）**
- 顶部条：返回、当天信息、确认穿搭
- 中央主区：角色预览与当前穿搭展示，优先保证角色主体完整可见
- 底部抽屉：分类标签条 + 服装网格
- 分类标签条可横向滚动，但 6 个类目必须全部进入数据源，不能因禁用而被过滤

这个布局把角色留在视觉中心，衣服从边缘区域被拿起、放到角色身上，和拖拽动作保持一致。顶部只放必要的导航和确认操作，避免抢走注意力；类目与网格承担浏览、筛选和选择，但不得遮挡角色主体超过 30%。

### Component Inventory

**顶部条**
- 返回按钮：图标按钮，返回主菜单
- 当天信息：文本，显示当前天数 / 今日标题
- 确认穿搭按钮：主操作按钮，提交当前搭配进入每日场景

**中央主区**
- 角色预览：主视觉，显示当前穿搭
- 穿搭命中区：拖拽落点区域，接收服装
- 当前选中提示：小型状态提示，显示已选服装或已装备状态

**底部抽屉**
- 分类标签栏：固定包含 `top`, `bottom`, `shoes`, `accessory`, `hair`, `makeup` 6 个类目；进度只决定 enabled/disabled，不决定是否显示
- 服装网格：卡片网格，展示该分类下的服装
- 单件服装卡片：显示服装本体、解锁状态、分类、选中/已装备状态
- 锁定标记：图标 + 文案，提示未解锁
- 拖拽提示：轻量反馈，提示可拖拽或可点击
- 空状态提示：当前分类无内容时显示

## States & Variants

| State / Variant | Trigger | What Changes |
|-----------------|---------|--------------|
| 默认状态 | 正常进入衣橱 | 显示分类、服装网格、角色预览与基本操作 |
| 加载状态 | `ProgressManager` / `TextureCache` 未就绪 | 显示轻量等待，不显示完整服装状态 |
| 空状态 | 当前分类没有服装 | 显示温和提示，不自动切换分类 |
| 锁定状态 | 物品尚未解锁 | 服装卡片显示锁定标记与解锁信息 |
| 资源异常状态 | 缩略图加载失败或 callback 返回 `null` | 卡片显示占位图，交互状态不被改变 |
| 数据异常状态 | `WardrobeDatabase` 返回无法渲染的物品数据 | 过滤异常条目并记录 warning；若该分类无可渲染卡片，显示空状态 |
| 依赖未就绪 / 超时状态 | `ProgressManager` 长时间未就绪 | 保持轻量 loading 或显示温和不可用提示，不渲染错误类目 |
| 已选中 / 已装备状态 | 玩家点击或装备某件服装 | 目标卡片与角色预览高亮 |
| 桌面端变体 | 鼠标悬停在卡片上 | hover 高亮更明显 |
| 触摸端变体 | 触摸操作 | 不依赖 hover，强调点击和拖拽反馈 |

## Interaction Map

| Component | Action | Input | Feedback | Outcome |
|---|---|---|---|---|
| 返回按钮 | 请求返回 | 点击 / 触摸 | 按钮轻微高亮，出现确认提示 | 玩家确认后返回主菜单；取消则留在衣橱 |
| 分类标签栏 | 切换 enabled 分类 | 点击 / 触摸 | 标签激活高亮，网格刷新 | 显示对应分类服装 |
| 禁用分类标签 | 查看锁定提示 | 点击 / 触摸 | 锁图标轻亮或轻缩 | 当前分类不变，网格不刷新 |
| 服装卡片 | 直接拖拽 | 拖拽 / 触摸拖动 | 卡片跟随、源卡 ghost、角色区高亮 | 落到角色上则应用该服装 |
| 服装卡片 | 点击备选 | 点击 / 触摸 | 卡片选中态 | 进入选中状态，等待应用 |
| 穿搭命中区 | 接收穿搭 | 拖拽落点 | 角色轻微强调 | 成功后更新当前穿搭 |
| 确认穿搭按钮 | 提交搭配 | 点击 / 触摸 | 按钮确认反馈 | 进入每日场景 |

拖拽是主路径：玩家从服装卡片直接拖到角色身上，作为最自然的换装方式。点击只是备选路径，用于玩家不方便拖拽时先选中再应用。

## Events Fired

| Player Action | Event Fired | Payload / Data |
|---|---|---|
| 点击返回按钮 | `ui.wardrobe.back_pressed` | `{ "current_day": day, "has_unsaved_changes": bool }` |
| 确认返回主菜单 | `ui.wardrobe.back_confirmed` | `{ "current_day": day }` |
| 取消返回主菜单 | `ui.wardrobe.back_cancelled` | `{ "current_day": day }` |
| 切换分类标签 | `wardrobe.category_pressed` | `{ "category": category }` |
| 点击禁用分类 | `wardrobe.category_locked_pressed` | `{ "category": category, "current_day": day }` |
| 点击服装卡片 | `wardrobe.item_pressed` | `{ "item_id": item_id, "category": category }` |
| 点击锁定服装 | `wardrobe.item_locked_pressed` | `{ "item_id": item_id, "unlock_day": unlock_day }` |
| 开始拖拽服装 | `wardrobe.item_drag_started` | `{ "item_id": item_id, "category": category }` |
| 拖拽成功装备 | `wardrobe.outfit_applied` | `{ "item_id": item_id, "category": category, "feedback_intensity": intensity }` |
| 点击确认穿搭 | `ui.wardrobe.confirm_pressed` | `{ "current_day": day, "equipped_items": equipped_items }` |
| 进入每日场景 | `scene.daily.entered` | `{ "day": day, "scene_id": scene_id }` |

返回、分类切换、物品点击、拖拽和确认穿搭都应发事件。确认穿搭会写入保存数据，因此这是一个需要特别注意的状态提交点，不只是普通 UI 反馈。

## Transitions & Animations

衣橱 UI 的转场应当轻快、柔和、不过度强调存在感。

- 进入衣橱：快速淡入，顶部条和底部抽屉先出现，角色区随后完成显示
- 离开衣橱：先收起底部抽屉，再整体淡出到主菜单或每日场景
- 切换分类：网格轻滑刷新，保持内容移动的连续感
- 拖拽服装：卡片跟随指尖移动，源卡保留 ghost，不做夸张飞行动画
- 装备成功：角色区轻微闪亮或呼吸式强调，不做爆闪
- 锁定 / 无效操作：只给很轻的提示反馈，不中断当前流程
- 空状态 / 加载状态：淡入淡出切换，减少突兀感

## Data Requirements

| Data | Source System | Read / Write | Notes |
|------|--------------|--------------|-------|
| 服装本体（名称、缩略图、分类） | 服装数据库 | Read | 服装卡片的基础内容 |
| 解锁状态 | 进度管理 | Read | 决定卡片是否锁定 |
| 全部 MVP 类目 | 服装数据库 / 衣橱 UI 常量 | Read | UI 始终显示 `top`, `bottom`, `shoes`, `accessory`, `hair`, `makeup` |
| 类目启用状态 | 进度管理 | Read | 决定分类是 enabled 还是 disabled；不得从 UI 中移除禁用类目 |
| 当前天数 | 进度管理 / 场景状态管理 | Read | 用于顶部显示和确认流程 |
| 当前搭配 | 拖拽换装 / 精灵分层渲染 | Read / Write | 权威 `equipped_items` 来自换装结果回写；衣橱 UI 只维护 transient selected/dragging 状态 |
| 缩略图资源 | 资源加载器 | Read | 网格卡片显示依赖项 |
| 角色当前穿搭 | 拖拽换装 / 精灵分层渲染 | Read | 用于中央预览和已装备状态 |
| 保存的穿搭上下文 | 保存/加载 | Read / Write | 确认穿搭时写入 `equipped_items` 与 `scene_in_progress` |
| 当天场景标识 | 场景/状态管理 | Read | 进入每日场景时使用 |

衣橱 UI 只消费这些系统提供的数据，不自己决定解锁规则、类目规则或保存规则。它可以写入“当前会话的搭配意图”，但真正的持久化和进度推进仍由保存/加载、进度管理和场景状态管理负责。

## Accessibility

- 所有主要操作都必须可通过触摸完成：返回、分类切换、卡片浏览、拖拽、确认穿搭都不能只依赖鼠标 hover。
- 每个可交互元素都要有清晰焦点态；键盘路径应按 `返回 → 分类标签 → 服装卡片网格 → 角色应用区域 → 确认穿搭` 的顺序移动。
- 服装状态不能只靠颜色区分，锁定、已选中、已装备必须同时使用图标、边框或角标。
- 拖拽必须有替代路径：玩家可以通过点击卡片再应用来完成同样结果。
- 所有可点击 / 可触摸目标都应足够大，避免误触，特别是底部抽屉里的分类标签和服装卡片。
- 动画要轻快，但需要保留减少动态效果的空间，避免过强的位移或闪烁。
- `hover`、`selected`、`equipped`、`keyboard_focus` 必须是可区分的视觉状态；键盘焦点不得被鼠标 hover 覆盖。
- 正文和关键 UI 文本应以 WCAG-AA 对比度作为临时基线，直到项目定义正式 accessibility tier。

## Localization Considerations

- 分类标签、按钮文案、锁定提示和空状态提示都必须走本地化文本，不在 UI 节点中硬编码。
- 分类标签建议中文 2-4 字，英文约 12 characters 内；翻译变长时标签栏需要支持横向滚动或自适应换行，但不能挤压服装网格。
- 服装名称卡片内建议中文 8 字以内，英文约 18 characters 内；超出时支持换行或缩放，避免覆盖缩略图、锁定图标或已装备角标。
- “第 N 天解锁”这类文本必须使用可本地化格式，不拼接固定中文字符串。
- 确认按钮建议中文 4-6 字，英文约 18 characters 内；返回按钮建议优先使用图标 + 可访问标签。空状态提示建议中文 12-18 字，英文约 40 characters 内。
- 所有布局关键文本应预留约 40% 文本扩展空间。
- 移动端底部抽屉是高风险区域：翻译后仍需保证分类标签、卡片、锁定标记和触摸热区不互相遮挡。

## Acceptance Criteria

- [ ] 衣橱 UI 从主菜单点击“开始今天”后在 500ms 内打开到可交互状态，或在依赖未就绪时显示轻量 loading。
- [ ] 衣橱 UI 从主菜单点击“开始今天”后正常打开，并显示当前天数、角色预览、分类标签和服装网格。
- [ ] 视口宽度 ≥768px 时，类目标签位于左侧，服装网格位于底部，角色居中且 UI 覆盖角色主体不超过 30%。
- [ ] 视口宽度 <768px 时，类目标签和服装网格位于底部区域，6 个类目均可通过横向滚动或等价方式访问。
- [ ] 衣橱 UI 始终显示 6 个 MVP 类目：`top`, `bottom`, `shoes`, `accessory`, `hair`, `makeup`；禁用类目显示锁定态而不是从 UI 中移除。
- [ ] 玩家点击或触摸分类标签时，服装网格刷新为对应分类，当前分类有明确激活状态。
- [ ] 玩家点击禁用分类标签时，当前分类不变，服装网格不刷新，并显示轻量锁定反馈。
- [ ] 玩家可以将已解锁服装卡片直接拖拽到角色区域，成功后角色预览更新，卡片显示已装备状态。
- [ ] 玩家可以通过点击服装卡片的备选路径完成选中 / 应用，不依赖拖拽。
- [ ] 未解锁服装显示锁定标记和解锁提示，且不能被拖拽或装备。
- [ ] 当前分类没有服装时，底部抽屉显示温和空状态，不崩溃、不自动跳转分类。
- [ ] 缩略图加载失败时，服装卡片显示占位图，卡片尺寸和交互状态不发生跳动。
- [ ] 触摸端可以完成返回、分类切换、卡片浏览、拖拽换装和确认穿搭。
- [ ] 玩家点击确认穿搭后，当前 `equipped_items` 被提交，并进入每日场景。
- [ ] 玩家按回退时先看到轻量确认提示；确认后返回主菜单且未确认编辑不保存，取消提示后仍停留在衣橱。
- [ ] 键盘焦点顺序为返回、分类标签、服装卡片网格、角色应用区域、确认穿搭，且 `keyboard_focus` 不与 hover/selected/equipped 混淆。
- [ ] 锁定、已选中、已装备等状态不能只通过颜色表达，必须有图标、边框或角标。
